## 1. Purpose and scope

Implement Layout A from the lockfile strategy design ([`2026-06-07_lockfile_strategy_design.md`](./2026-06-07_lockfile_strategy_design.md)) and close the **dependency-file reproducibility** dimension of the open reproducibility gate inventoried in [`2026-06-07_dependency_reproducibility_inventory.md`](./2026-06-07_dependency_reproducibility_inventory.md).

This patch is **dependency-file reproducibility only**:

- create `requirements.in` (runtime direct-dependency source of truth with every direct dep pinned);
- regenerate `requirements.txt` as a `pip-tools`-compiled lockfile with full transitive pins and SHA256 hashes;
- create `requirements-dev.txt` for dev/test/tooling deps (moves `httpx` out of runtime; declares `pip-tools` and `pytest`);
- record the implementation, validation, and follow-up gaps.

**Out of scope (held for later patches):**

- Explicit Dockerfile `--require-hashes` flag (hash-strategy posture **H4** from the design — deferred to a one-step follow-up after the first successful Render build under the lockfile). The compiled `requirements.txt` does carry hashes, and `pip install -r requirements.txt` verifies them at install time even without the explicit flag.
- Docker base-image digest pin (held for **11B-b3**).
- GitHub Actions SHA pinning (held for **11B-b4**).
- `anchor-retention-prune.yml` fix-or-delete (held for **11B-b5**).
- `alembic` dead-weight removal (held for **11B-b6**).
- `httpx<2` deprecation hygiene (optional **11B-b7**).
- Any deploy decision.

### 5.2 Hash posture (H4)

The lockfile carries SHA256 hashes for every resolved distribution. **`pip install -r requirements.txt` enters hash-checking mode automatically when every entry in the file carries hashes** — every wheel/sdist downloaded is verified against the recorded hash before install, on both the operator workstation and the Render builder. The Dockerfile was **not changed** in this patch; the explicit `--require-hashes` flag is **deferred to a later patch** (the flag's only added strictness is to refuse a file that contains *any* un-hashed entries — the implicit hash-checking already in effect against today's fully-hashed lockfile is the same per-wheel verification). The deferred flip is held for a one-step follow-up after the first successful Render rebuild under the lockfile, as designed (§4.2 of the strategy design). Today's install path (`pip install --no-cache-dir -r requirements.txt`) continues to work unchanged and now verifies hashes at install time.

## 6. Dev requirements (`requirements-dev.txt`)

Three lines:

```
pip-tools
pytest
httpx==0.28.1
```

- `pip-tools` and `pytest` are left **unpinned** intentionally so the operator can re-install on any 3.11 environment without re-compiling; the implementation patch's installed versions were `pip-tools==7.5.3` and `pytest==9.0.3`. Pinning these is held for a later dev-tooling hygiene patch.
- `httpx==0.28.1` is the version observed in the Patch 11B-b1 inventory and the version the test suite was written against. The CI workflows (`anchor-rate-limit-ci.yml`) install `requirements.txt` first; the Render image does **not** install `requirements-dev.txt`. This change therefore takes `httpx` out of the production install while keeping the test-side `fastapi.testclient` path working in CI.
- The `requirements-dev.txt` file itself is **not** a hashed lockfile — it is a developer-facing pip requirements file, deliberately simple (the design records this).

## 7. Validation

All validation steps executed inside a freshly created Python 3.11 venv (`.venv-lockcheck`) on the operator workstation. `--trusted-host pypi.org --trusted-host files.pythonhosted.org` was passed at `pip install` time as a workstation-local workaround for the same TLS path issue documented in earlier patches (Patch 11B-a, 11C). The lockfile itself does **not** carry any trusted-host directive — those were stripped from the autogenerated header.

## 8. `pip-audit` follow-up

Local `pip-audit -r requirements.txt` was attempted inside the verification venv with `pip-audit==2.10.0`. The tool's first action is to upgrade `pip`/`wheel`/`setuptools` inside an isolated temporary venv to satisfy its own resolver. That upgrade reaches PyPI over a network path subject to the **same workstation TLS certificate-verification failure** recorded in:

- [`2026-06-07_pip_audit_scan.md`](./2026-06-07_pip_audit_scan.md) (Patch 11B-a) — first inconclusive local pip-audit run, TLS path;
- [`2026-06-07_pyjwt_remediation.md`](./2026-06-07_pyjwt_remediation.md) (Patch 11C) — second inconclusive local pip-audit run, TLS path;
- [`2026-06-07_fastapi_starlette_remediation.md`](./2026-06-07_fastapi_starlette_remediation.md) (Patch 11D-b) — third inconclusive local pip-audit run, TLS path.

The pip-audit subprocess does **not** honour the workstation `--trusted-host` workaround used elsewhere in this patch (the subprocess builds its own pip invocation). Result: **local pip-audit run inconclusive — workstation TLS path.** This is identical in shape and root cause to the three prior local pip-audit attempts.

**Follow-up required after commit/push:** manually trigger the existing `Anchor Dependency Audit (pip-audit)` workflow (`.github/workflows/dependency-audit.yml`) against the new lockfile and confirm that the scanned set remains PASS. The workflow already audits `requirements.txt`; no workflow change is needed.
